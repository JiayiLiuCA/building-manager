## 1. 业务背景与核心变化

客户是物业公司,管理对象从住宅小区变为**产业园区**,缴费主体由个人业主变为**入驻企业**(公司客户)。整体仍是给客户看的演示 demo。

| 维度 | 旧(小区) | 新(产业园) |
|---|---|---|
| 服务对象 | 户/业主(Household/Resident) | 入驻企业(Company),整栋独占一栋楼,或占用某楼栋内的若干楼层/单元 |
| 物业端角色 | 单一 admin | 主管(supervisor)+ 客服(cs),客服按「管辖企业名单」划分数据权限 |
| 欠费与催缴 | 核心卖点 | **弱化**(企业欠费少),改造为「收款跟进」;新增缴费习惯档案与发票管理 |
| 物业端导航 | 驾驶舱/日报/工单投诉/催缴/缴费钻取/户档案 | 驾驶舱/日报 + **经营管理、服务品质、内控管理**三大类及子模块 + 通知管理/企业档案/权限设置 |

空间层级:园区(1 个,占地一大片)→ **区(A 区 / B 区 / C 区)** → 楼栋(每区若干栋)→ 楼层/单元。企业入驻形态两种并存,数据模型与所有楼栋维度的逻辑必须同时支持:①**整栋独占**——一家企业占有一整栋楼(该楼栋的收缴率即该企业的收缴率);②**多户楼栋**——一栋楼内多家企业,各占若干楼层/单元。原「空置户」完整链路(登记空置、半价、停水停电待办、⚪️核实建议)**整体移除**,相关 types / store 动作 / selectors / 页面一并清理。

### 2.2 权限规则(全局适用,必须严格执行)

- `Role` 改为 `'supervisor' | 'cs' | 'company'`;新增实体 `CsAssignment { csUsername: string; companyIds: string[] }`。
- 提供**唯一入口** selector `visibleCompanyIds(state)`:主管 → 全部企业;客服 → 被分配的企业;企业账号 → 仅自己。
- **所有**与企业相关的数据读取(账单、收缴、减免、欠费、工单、投诉、满意度、发票、通知触达、收款跟进、驾驶舱、日报……)一律先经该 selector 过滤再聚合。客服界面上出现的每一个数字都只能来自其名下企业。
- 内控管理(巡检/核抄/任务清单)是物业内部数据,与企业无关但与人有关:主管看全部;客服只看指派给自己的记录/任务;企业端完全不可见。
- 主管专属「权限设置」页:列出全部客服账号,为每个客服勾选管辖企业,保存即生效——同一会话内切到客服账号,其驾驶舱/日报/各列表立即随之变化(这是重点演示链路之一)。
- 客服发布通知时,影响范围只能选择自己管辖的企业或其所在的区/楼栋。

## 4. 驾驶舱与日报(按当前登录角色的可见数据生成)

### 4.1 总原则

- **数字显性直接展示**:关键指标一律数字卡片(KpiCard);所有 Recharts 图表必须**常显数值标签**(LabelList 或自定义 label),并在图旁/图下配数据明细小表;**不允许**任何"点击/hover 后才能看到数值"的设计。曲线图、柱状图本身不够直接,必须叠加显性数字。
- 主管与客服使用同一套驾驶舱/日报组件,但全部指标经各自 `visibleCompanyIds` 聚合;主管与两位客服三者的数字应肉眼可辨地不同(由 seed 编排保证)。

### 4.2 驾驶舱内容

1. **顶部 KPI 卡**:本月应收 / 实收 / 收缴率;**费用减免数据**(本月与年度累计减免金额、减免占应收比);本月工单(新增数 / 关单率);本月整体满意度。
2. **欠费数据(弱化处理)**:一张放在次要位置的紧凑卡片或折叠区——历史欠费 + 当期应收欠费合计金额、涉及企业数,点击展开明细。数字仍显性,但不再是首屏视觉焦点。
3. **图表区**(四张,全部带显性数值 + 明细):收费趋势(近 12 月应收 vs 实收)、收缴对比(按 A/B/C 区,柱上直接标数值)、工单状态分布、企业满意度分布。
4. **缴费收款板块**:
   - 按**楼栋**收缴率表:按区分组列出各楼栋(区 / 楼栋 / 入驻企业数 / 应收 / 实收 / 收缴率),含**区小计**行;整栋独占企业所在楼栋的收缴率即该企业收缴率;
   - **客服个人**收缴率表:客服 / 名下企业数 / 应收 / 实收 / 收缴率(客服有明确的服务区域=管辖企业划分);
   - 两表底部均有**合计行**做数据汇总。主管见全量;客服只见与自己相关的行。
5. **收款跟进(原催缴,整体弱化)**:对未缴清企业逐一给出**前置判断**,判断依据三项并列展示:
   - **缴费习惯**(如"习惯每月 25 日对公转账,未到付款日 → 暂不跟进");
   - **报事报修关单情况**(有超时未闭环工单 → 暂缓);
   - **投诉关单情况**(有未闭环投诉 → 暂缓)。
   沿用并改造现有三色建议机制(🟢建议跟进 / 🟠暂缓 / ⚪️待沟通核实);每家企业行内展示其**历史减免记录**(月份、费类、金额、原因)。驾驶舱只展示最需关注的前几家,完整列表在「企业档案 → 收款跟进」Tab。

### 4.3 日报

改为由 selectors 从当前角色**可见数据实时生成**(替代纯手写文案):今日收款与收缴率变化、新增/关闭工单、维保执行、巡检完成、发布的通知、新增满意度评价、收款跟进动态、"今天是 XX 企业的习惯付款日"提示。保留 AI 摘要文案层(`mock-content` 全部改写为园区语境)。主管与客服的日报数字各自成立,且与各自驾驶舱一致。

## 5. 经营管理(四个子模块,页面骨架统一)

统一骨架:顶部**年 / 季 / 月**维度切换 + 期间选择器;KPI 卡(达成金额、目标金额、达成率);趋势图(带显性数值);明细表。seed 需为各费类按期间生成**目标值**用于计算达成率。

1. **物业服务收费**:年/季/月收费达成金额及**达成率**;支持下钻:园区 → 区 → 楼栋 → 企业(改造现有缴费钻取多级页;整栋独占的楼栋下钻后直达该企业)。
2. **车辆服务收费**:按**固定车位收入 / 临时停放车辆收入 / 租赁车位收入**三个口径,展示各口径与合计的年/季/月达成金额及趋势(多序列图 + 分口径 KPI)。
3. **水电能耗收费**:按**购水 / 购电**两个口径,展示年/季/月收费金额及趋势。
4. **增值服务收入**:按业务(**合同类型**)展示达成金额及趋势;合同类型:**到家服务、资产运营、零售服务**(名称按客户文档执行,如需贴合园区语境仅可在展示文案层微调,结构不变)。seed 生成若干增值服务合同(企业、类型、金额、期间)。

## 6. 服务品质(三个子模块)

1. **维修工单**:类型细分为**企业报事报修**与**公共区域维修**两类;沿用现有事件源工单模型与完整处理流(报修→接单→派单→预约→完工→签字关单→评价);指标:响应及时率、完成及时率、**关单率**;提供**月度集成分析**视图(本月与近 12 月整体维修数据:量、及时性、关单率、类型分布)。投诉保留现有完整链路(派单、回复、申请主管介入、关闭),作为本页的「投诉」Tab。公共区域维修工单可一键生成关联通知(见 §8.1)。
2. **维保工单**:分**消防维保 / 电梯维保 / 日常维保**三类;基于**维保计划**生成(计划时间 vs 实际执行时间);指标:及时性、关单率、计划完成率;月度集成分析整体维保数据。
3. **客户满意度**:两个来源——
   - **被动满意度**:企业报事报修关单时附加的满意度评价,自动抓取(现有 RATED 事件);
   - **满意度调研**:线上开展;物业端发起调研(制式满意度调研表),企业端填写;seed 预置历史两期调研结果。
   月度体现整体满意度数据:总体分、来源拆分、近 12 月趋势、低分评价明细。

## 7. 内控管理(三个子模块;物业内部,企业端不可见)

制式表单模板后续由物业公司提供;当前按下述合理结构自行设计,字段结构做成易替换的配置(集中定义在 seed/constants 或独立 config)。

1. **日常巡检**:制式巡检表单——巡检区域、细分巡检项 checklist(安防、消防、保洁、设备等)、结果(合格/异常)、异常描述、照片占位、巡检人、计划时间/实际时间;指标:巡检完成率、及时率、异常项数(体现**巡检质量与时效性**)。
2. **能耗核抄**:分**水表核抄 / 电表核抄**两大类制式表单——表编号、位置/归属、上期示数、本期示数、本期用量、核抄人、核抄时间;**月度用量趋势 + 同比/环比**数据分析(图表带显性数值 + 数字表)。
3. **工作任务清单**:制式表单;任务具备**年 / 季 / 月 / 周**四级层级并支持**穿透下钻**(年度目标 → 季度拆解 → 月度任务 → 周任务);指标:各层级**达成率与及时性**;任务有责任人(主管或客服),客服只见自己的任务,主管见全部。

## 8. 新增功能

### 8.1 通知管理(主管、客服均可发布)

- 通知类型:**公共区域维修、停水通知、停电通知**、一般公告。
- 字段:标题、类型、内容、影响范围(全园区 / 指定区 / 指定楼栋 / 指定企业)、生效时间段、发布人、发布时间、状态(生效中/已过期/已撤销)。
- 物业端:列表 + 新建 + 撤销;客服可选范围受管辖限制(§2.2);可从公共区域维修工单一键带出预填通知。
- 企业端:**首页顶部醒目展示**与自己相关的生效中通知(按影响范围过滤),可查看历史通知。

### 8.2 企业档案与缴费习惯

- 「企业档案」列表(scoped,可按区/楼栋筛选)→ 企业详情:企业名称、行业、入驻位置(区 / 楼栋 / 整栋或楼层单元)、面积、联系人/电话、合同期、状态;账单与减免记录;工单与投诉;发票;收款跟进记录。
- **缴费习惯**(重点):每月付款日、付款方式(对公转账 / 支票 / 线上支付等)、备注;客服/主管可编辑。企业欠费情况少,demo 重心从"催"转向"记录习惯、按习惯跟进"。
- 缴费习惯的三个消费场景:驾驶舱收款跟进的前置判断(§4.2.5)、企业档案展示、日报付款日提示(§4.3)。
- **历史减免记录**:每笔含月份、费类、减免金额、原因;在企业详情与收款跟进中展示;驾驶舱减免 KPI 由其聚合(§4.2.1)。

### 8.3 发票管理

- 月度发票,按 **企业 × 月份 × 费类** 组织;费类即经营管理四大类:物业服务费 / 车辆服务费 / 水电能耗费 / 增值服务费。
- **客服/主管上传**:mock 实现——选择本地文件仅记录文件名与元数据入内存 store;同时在 `public/invoices/` 预置 2–3 个示例 PDF 用于下载演示。字段:企业、月份、费类、金额、文件名、上传人、上传时间。
- **企业端「发票查询」**:按月份/费类筛选,查看与下载自己的发票;seed 预置每家企业近几个月的发票记录,保证两端一打开就有数据。

## 9. 企业端(原业主端改造)

导航:**首页**(生效通知 + 本月账单概览 + 待办)、**报事报修**(含投诉,沿用现有流程与电子签字)、**账单与缴费**(四大费类账单、缴费流程沿用;催缴弹窗弱化为仅在物业发起跟进时出现)、**发票查询**、**满意度调研**(填写物业发起的调研问卷)、**AI 咨询**(保留,话术与规则改园区语境)、**企业信息**(基本信息 + 缴费习惯,只读)。

## 10. Seed 编排(演示故事线)

- 园区 1 个(「和美产业园」),分 **A 区 / B 区 / C 区**,每区 4–6 栋、全园区约 13–15 栋;企业共 **30 家**:约 10 家**整栋独占**(一企一楼),其余约 20 家分布在若干**多户楼栋**中(各占楼层/单元)。其中 **3 家故事企业**拥有登录账号(见下),其余 27 家为背景企业——沿用现有 `generators` 固定种子模式批量生成,无登录账号,但完整参与物业端全部列表、聚合与图表展示:
  - **企业①(A 区 A3 栋 · 整栋独占,科技公司)**:模范客户——习惯每月 5 日对公转账、按时缴清、满意度高;演示"未到付款日 → 暂不跟进"。
  - **企业②(B 区 B2 栋 3–4 层,精密制造)**:所在楼栋为多户楼栋;有历史减免记录 + 当月一张超时未闭环报修工单与一条关联投诉 → 收款跟进"🟠暂缓";物业完结工单 + 闭环投诉后**实时变为 🟢**(复刻现 demo 的实时联动核心卖点)。
